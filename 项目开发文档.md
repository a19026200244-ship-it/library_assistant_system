# 智能图书馆助手项目开发文档

## 1. 项目概述

### 1.1 项目名称
智能图书馆助手（Library Assistant System）

### 1.2 项目背景
传统图书馆信息查询通常依赖人工咨询、固定检索终端或分散的规章页面，存在以下问题：

1. 学生需要花费较多时间查找图书位置；
2. 图书馆开放时间、借阅规则、楼层分布等信息分散，检索效率低；
3. 普通检索系统多为关键词匹配，无法很好理解自然语言问题；
4. 交互方式较单一，体验不够友好。

为解决以上问题，本项目设计并实现了一个面向校园场景的智能图书馆助手。系统融合了大语言模型、RAG 检索增强、工具调用、语音播报和 Web 前端交互能力，使用户能够通过自然语言快速完成图书查询与馆内信息咨询。

### 1.3 项目目标
本项目的目标如下：

- 构建一个可进行自然语言问答的图书馆智能助手；
- 支持图书位置查询、分类查询、在馆状态查询、图书推荐、馆内规章问答；
- 通过 RAG 机制提升规则类问答的准确性；
- 通过工具调用提升结构化查询能力；
- 提供 Web 前端，方便教师和同学直接演示与测试；
- 支持语音播报，提升交互体验；
- 具备一定的对话记忆能力，使多轮对话更自然。

### 1.4 适用对象
- 在校学生
- 图书馆管理人员
- 课程项目验收教师
- 需要演示智能问答系统的开发者

## 2. 项目功能概述

### 2.1 已实现核心功能

#### （1）图书位置查询
用户可直接输入书名或作者名，系统返回图书位置、作者、状态等信息。

示例：
- “《算法导论》在哪？”
- “科曼的书在哪里？”

#### （2）按分类查询图书
用户可按主题、学科方向或分类关键词查询图书。

示例：
- “图书馆有什么科技类书籍？”
- “推荐几本计算机方面的书”

#### （3）按状态查询图书
用户可查询某类图书是否在馆、是否已借出。

示例：
- “哪些计算机书被借出了？”
- “现在有哪些书在馆？”

#### （4）图书推荐
系统可以根据关键词、专业方向或兴趣主题推荐相关图书。

示例：
- “推荐几本人工智能相关的书”
- “给我推荐几本考研计算机书籍”

#### （5）图书馆规章制度问答
用户可查询开放时间、借阅规则、馆内要求、楼层布局等信息。

示例：
- “图书馆几点开门？”
- “借书最多能借几本？”
- “3 楼都有什么区域？”

#### （6）多轮对话能力
系统保留最近若干轮对话历史，使用户可以在上下文中连续提问。

示例：
- 用户：算法导论在哪？
- 助手：在 3 楼 L3-C01 架
- 用户：那它的作者是谁？
- 助手：能够结合上文理解“它”指代《算法导论》

#### （7）语音播报能力
系统支持将文字回答转换为语音并播放，增强演示效果与交互体验。

#### （8）Web 页面展示
项目提供基于 `HTML + CSS + JavaScript` 的前端页面，用于：

- 展示聊天对话；
- 渲染图书查询卡片；
- 通过输入框提交问题；
- 使用麦克风按钮进行语音输入；
- 通过朗读按钮播放回答语音。

## 3. 技术方案与总体架构

### 3.1 技术栈

#### 后端技术
- Python
- FastAPI
- OpenAI SDK（用于调用 DeepSeek 接口）
- LangChain
- FAISS
- HuggingFace Embeddings
- Edge-TTS

#### 前端技术
- HTML5
- CSS3
- JavaScript
- Font Awesome
- Marked.js（用于渲染 Markdown 内容）

#### 模型与能力
- DeepSeek Chat：负责对话理解、推理与工具调用
- BAAI/bge-small-zh-v1.5：负责文本向量化
- FAISS：负责向量检索
- Edge-TTS：负责文本转语音

### 3.2 系统总体架构

系统整体采用“前后端分离 + Agent 工具调用 + RAG 检索增强”的架构。

#### 架构分层

1. 表现层  
   由 `index.html` 构成，负责用户界面展示、输入采集、消息渲染、语音触发。

2. 接口层  
   由 `app.py` 提供 REST API，包括聊天接口和 TTS 接口。

3. 业务层  
   由 `main.py` 完成 Agent 核心逻辑，包括：
   - 对话消息组装
   - 工具调用
   - 对话记忆管理
   - 最终答案生成

4. 知识层  
   包括：
   - `data/library_books.json`：结构化图书数据
   - `data/library_knowledge.txt`：图书馆规章文本
   - `data/faiss_library_index/`：向量数据库索引

5. 语音层  
   由 `tts.py` 提供 TTS 音频生成功能。

### 3.3 系统工作流程

#### 文本问答流程
1. 用户在前端输入问题；
2. 前端调用 `/api/chat`；
3. 后端读取历史消息并组装为完整上下文；
4. DeepSeek 判断是否需要调用工具；
5. 若需调用工具，则执行对应函数；
6. 若为规章类问题，则通过向量库检索相关知识片段；
7. 模型结合工具结果或 RAG 结果生成最终回答；
8. 后端将答案和图书元数据返回前端；
9. 前端渲染聊天消息和图书卡片。

#### 语音播报流程
1. 用户点击朗读按钮；
2. 前端调用 `/api/tts`；
3. 后端使用 Edge-TTS 将文本合成为 MP3；
4. 浏览器接收音频流并播放。

#### 对话记忆流程
1. 前端维护 `chatHistory`；
2. 每次请求时携带历史消息；
3. 后端截取最近若干轮对话；
4. 组合为 `system + history + current user message`；
5. 模型据此完成上下文理解。

## 5. 核心模块设计

### 5.1 配置模块设计

配置模块统一写在 `config.py` 中，其作用是减少硬编码，方便部署与维护。

主要配置项如下：

- `DEEPSEEK_API_KEY`：调用大模型接口所需密钥
- `DEEPSEEK_BASE_URL`：模型服务地址
- `DEEPSEEK_MODEL`：所使用的模型名称
- `EMBEDDING_MODEL_NAME`：嵌入模型名称
- `EMBEDDING_DEVICE`：嵌入模型运行设备
- `CHUNK_SIZE`：文本切片大小
- `CHUNK_OVERLAP`：切片重叠长度
- `SIMILARITY_TOP_K`：向量检索返回条数
- `TTS_VOICE`：语音角色
- `MAX_HISTORY_ROUNDS`：保留对话轮数

### 5.2 数据加载模块设计

系统启动时会加载两类核心数据：

1. 图书结构化数据  
   从 `library_books.json` 中读取，加载为 Python 列表，供工具函数检索。

2. 规则知识向量库  
   通过 FAISS 加载本地索引，供规章类问题检索使用。

### 5.3 工具模块设计

为提升图书查询的准确性，系统没有仅依赖大模型自由回答，而是额外定义了多个工具函数。

#### 已实现工具

1. `query_book_location(book_name)`
   - 功能：按书名或作者查找图书
   - 返回：图书列表 JSON

2. `query_books_by_category(category)`
   - 功能：按分类、主题或描述检索图书
   - 返回：图书列表 JSON

3. `query_books_by_status(status)`
   - 功能：按图书状态查询
   - 返回：图书列表 JSON

4. `recommend_books(keyword)`
   - 功能：根据兴趣或主题推荐图书
   - 返回：图书列表 JSON

5. `query_library_rules(question)`
   - 功能：基于 RAG 查询图书馆规章和馆内信息
   - 返回：检索到的文本片段

### 5.4 Agent 模块设计

Agent 是本项目最核心的设计部分。其主要思路是：

1. 使用系统提示词约束角色与回答风格；
2. 将工具描述注册给大模型；
3. 由模型自主判断是否要调用工具；
4. 执行工具后，再次调用模型生成自然语言答案。

#### Agent 核心流程

```text
用户问题
   ↓
拼接 system prompt + 历史消息 + 当前问题
   ↓
调用 DeepSeek
   ↓
是否触发工具调用？
   ├── 否 → 直接返回模型回答
   └── 是 → 执行工具函数
               ↓
           追加 tool message
               ↓
        再次调用模型生成最终答案
```

### 5.5 对话记忆设计

为了支持多轮对话，系统在后端使用 `_trim_chat_history()` 对历史消息进行截断处理，仅保留最近若干轮，以避免上下文无限增长。

这样设计具有两个好处：

1. 让模型保持上下文连贯，能够理解“它”“这本书”等指代；
2. 控制上下文长度，降低推理成本。

### 5.6 RAG 模块设计

RAG（Retrieval-Augmented Generation，检索增强生成）用于处理规章制度、开放时间、楼层分布等文本型问题。

#### 实现过程

1. 读取 `library_knowledge.txt`；
2. 使用 `CharacterTextSplitter` 进行文本切分；
3. 使用 `HuggingFaceEmbeddings` 对每个文本片段向量化；
4. 使用 `FAISS` 构建本地向量索引；
5. 用户提问时，从向量库中检索最相关片段；
6. 将检索结果交给大模型组织成自然语言回答。

#### RAG 的意义

- 减少模型“编造答案”的概率；
- 提升馆内规章类问答的可靠性；
- 方便后续扩展 PDF、文档库等更多知识源。

### 5.7 语音模块设计

#### TTS 语音播报
项目采用 `Edge-TTS` 实现文本到语音的转换。

具体流程：

1. 前端发送文本到 `/api/tts`；
2. 后端调用 `tts.py` 生成临时音频文件；
3. 通过 `FileResponse` 将音频返回给浏览器；
4. 浏览器通过 `Audio` 对象播放；
5. 后端利用 `BackgroundTask` 自动清理临时文件。

#### 语音输入说明
当前前端页面通过浏览器内置语音识别接口采集语音输入，适合演示与轻量使用。  
如果后续继续升级为 Whisper 后端识别方案，可将浏览器录音文件上传至服务端，再由模型进行转写。

### 5.8 前端模块设计

前端采用单页 HTML 架构，核心功能包括：

- 顶部导航栏展示系统身份；
- 中部聊天区域展示消息；
- 底部输入区域负责文本输入、发送和录音；
- 针对图书查询结果渲染图书卡片；
- 支持对 Markdown 回答的样式化展示；
- 支持语音朗读按钮；
- 支持滚动到底部、加载状态、错误状态等交互细节。

相较于 Gradio，当前方案具有更高的页面自由度、更适合产品化展示，也更方便后续接入学校项目答辩场景。

## 8. 关键实现思路

### 8.1 为什么采用“工具调用 + RAG”混合方案

本项目将问题分成两类：

1. **结构化查询问题**  
   如图书位置、作者、状态、分类、推荐等。这类问题适合走工具调用，因为数据结构明确、结果稳定、准确率高。

2. **非结构化知识问题**  
   如开放时间、馆内规则、FAQ 等。这类问题适合走 RAG，因为回答依赖文本语义理解。

因此，系统采用混合方案：

- 结构化问题 → 工具调用
- 规章类问题 → RAG 检索
- 最终统一交给大模型组织语言输出

这种方案兼顾了准确性、灵活性和可扩展性。

### 8.2 为什么使用 FastAPI

选择 FastAPI 的原因：

- 编写接口简洁；
- 自带数据校验；
- 性能较好；
- 适合快速搭建 AI 应用 API；
- 方便与前端页面对接；
- 方便后期扩展成多接口服务。

### 8.3 为什么使用 HTML + CSS + JS，而不是继续使用 Gradio

Gradio 适合快速原型验证，但在正式展示中存在局限：

| 对比项 | Gradio | HTML + CSS + JS |
| --- | --- | --- |
| 搭建速度 | 快 | 中等 |
| 页面自由度 | 一般 | 很高 |
| UI 美观度 | 受组件限制 | 可深度定制 |
| 交互细节 | 组件化为主 | 可完全自定义 |
| 产品化展示 | 一般 | 更强 |
| 后续扩展 | 受框架约束 | 更灵活 |

因此，本项目最终选择将前端改造为原生 Web 页面，以获得更好的演示效果与可控性。

## 9. 环境配置与运行方式

### 9.1 开发环境

建议环境如下：

- Python 3.10 及以上
- Windows / Linux / macOS
- 已联网环境（首次拉取模型或调用外部服务时需要）

### 9.2 依赖安装

```bash
pip install -r requirements.txt
```

### 9.3 环境变量配置

根据 `.env.example` 配置模型 API Key，例如：

```bash
set DEEPSEEK_API_KEY=你的密钥
```

或在 Linux/macOS 中：

```bash
export DEEPSEEK_API_KEY=你的密钥
```

### 9.4 构建向量库

首次运行前，需要先构建知识库向量索引：

```bash
python ingest.py
```

### 9.5 启动项目

```bash
python app.py
```

浏览器访问：

```text
http://localhost:8000
```

### 9.6 Docker 部署

项目支持 Docker 部署，可通过以下命令启动：

```bash
docker-compose up -d --build
```

- 询问楼层布局；
- 询问 FAQ 类问题。

#### 多轮对话测试
- 先问图书名称；
- 再追问“它在哪一层”“作者是谁”等；
- 检查是否能够正确理解代词指代。

#### TTS 测试
- 点击朗读按钮；
- 检查是否能正常播放语音；
- 检查音频文件是否在后台被自动清理。

#### 前端交互测试
- 输入框发送是否正常；
- 回车发送是否正常；
- 加载状态是否显示；
- 图书卡片是否渲染；
- 长文本滚动是否正常。

### 10.2 异常情况测试

- API Key 未配置；
- 向量库未构建；
- 知识文件缺失；
- 模型服务不可用；
- 浏览器不支持语音输入。

系统在这些情况下应尽量给出可理解的错误提示，而不是直接崩溃。

## 11. 项目亮点与创新点

### 11.1 场景明确
项目聚焦“校园图书馆”这一实际应用场景，问题明确、用户对象清晰、功能实用性强。

### 11.2 混合式问答架构
系统不是单纯调用大模型，而是结合：

### 11.5 前端产品化展示

### 11.6 具备扩展潜力
本项目后续可以继续扩展为：

- 图书借阅状态管理系统；

- 多知识源 RAG 系统；
- 校园级综合问答助手。
